## <a name="encryption-and-transport-protocols"></a><span data-ttu-id="7ccc6-128">加密及傳輸通訊協定</span><span class="sxs-lookup"><span data-stu-id="7ccc6-128">Encryption and transport protocols</span></span>

<span data-ttu-id="7ccc6-129">最好先從兩個層面考量 PowerShell 遠端連線的安全性：初始驗證，以及進行中的通訊。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-129">It is helpful to consider the security of a PowerShell Remoting connection from two perspectives: initial authentication, and ongoing communication.</span></span>

### <a name="initial-authentication"></a><span data-ttu-id="7ccc6-131">初始驗證</span><span class="sxs-lookup"><span data-stu-id="7ccc6-131">Initial authentication</span></span>

<span data-ttu-id="7ccc6-132">驗證會確認用戶端至伺服器的身分識別，在理想的情況下，也會確認伺服器到用戶端的身分識別。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-132">Authentication confirms the identity of the client to the server - and ideally - the server to the client.</span></span>

<span data-ttu-id="7ccc6-133">當用戶端使用其電腦名稱連線到網域伺服器時，預設的驗證通訊協定是 [Kerberos](/windows/win32/secauthn/microsoft-kerberos)。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-133">When a client connects to a domain server using its computer name, the default authentication protocol is [Kerberos](/windows/win32/secauthn/microsoft-kerberos).</span></span> <span data-ttu-id="7ccc6-134">Kerberos 可保證使用者識別與伺服器識別，而不會傳送任何種類可重複使用的認證。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-134">Kerberos guarantees both the user identity and server identity without sending any sort of reusable credential.</span></span>

<span data-ttu-id="7ccc6-135">當用戶端使用其 IP 位址連線到網域伺服器時，或連線到工作群組伺服器時，便無法進行 Kerberos 驗證。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-135">When a client connects to a domain server using its IP address, or connects to a workgroup server, Kerberos authentication is not possible.</span></span> <span data-ttu-id="7ccc6-136">在此情況下，PowerShell 遠端會仰賴 [NTLM 驗證通訊協定](/windows/win32/secauthn/microsoft-ntlm)。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-136">In that case, PowerShell Remoting relies on the [NTLM authentication protocol](/windows/win32/secauthn/microsoft-ntlm).</span></span> <span data-ttu-id="7ccc6-137">NTLM 驗證通訊協定可保證使用者識別，且不會傳送任何種類可委派的認證。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-137">The NTLM authentication protocol guarantees the user identity without sending any sort of delegable credential.</span></span> <span data-ttu-id="7ccc6-138">為了要證明使用者識別，NTLM 通訊協定需要用戶端與伺服器從使用者的密碼來計算工作階段金鑰，而不會交換密碼本身。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-138">To prove user identity, the NTLM protocol requires that both the client and server compute a session key from the user's password without ever exchanging the password itself.</span></span> <span data-ttu-id="7ccc6-139">伺服器通常不知道使用者的密碼，因此會和知道使用者密碼的網域控制站互相通訊，並為伺服器計算出工作階段金鑰。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-139">The server typically does not know the user's password, so it communicates with the domain controller, which does know the user's password and calculates the session key for the server.</span></span>

<span data-ttu-id="7ccc6-140">但 NTLM 通訊協定並不能保證伺服器識別。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-140">The NTLM protocol does not, however, guarantee server identity.</span></span> <span data-ttu-id="7ccc6-141">因為所有通訊協定都使用 NTLM 進行驗證，所以可以存取已加入網域之電腦的電腦帳戶之攻擊者，就能叫用網域控制站計算出 NTLM 工作階段金鑰，然後模擬該伺服器。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-141">As with all protocols that use NTLM for authentication, an attacker with access to a domain-joined computer's machine account could invoke the domain controller to compute an NTLM session-key and thereby impersonate the server.</span></span>

<span data-ttu-id="7ccc6-142">預設會停用 NTLM 的驗證，但在目標伺服器上設定 SSL 或在用戶端設定 WinRM TrustedHosts 設定時，可能會允許。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-142">NTLM-based authentication is disabled by default, but may be permitted by either configuring SSL on the target server, or by configuring the WinRM TrustedHosts setting on the client.</span></span>

#### <a name="using-ssl-certificates-to-validate-server-identity-during-ntlm-based-connections"></a><span data-ttu-id="7ccc6-143">NTLM 連線期間使用 SSL 憑證來驗證伺服器識別</span><span class="sxs-lookup"><span data-stu-id="7ccc6-143">Using SSL certificates to validate server identity during NTLM-based connections</span></span>

<span data-ttu-id="7ccc6-144">由於 NTLM 驗證通訊協定無法確保目標伺服器 (只有它已經知道您的密碼) 的身分識別，所以您可以將目標伺服器設定為使用 SSL 進行 PowerShell 遠端。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-144">Since the NTLM authentication protocol cannot ensure the identity of the target server (only that it already knows your password), you can configure target servers to use SSL for PowerShell Remoting.</span></span>
<span data-ttu-id="7ccc6-145">將 SSL 憑證指派至目標伺服器 (如果由用戶端也信任的憑證授權中心發出)，可讓 NTLM 驗證能保證使用者識別與伺服器識別。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-145">Assigning a SSL certificate to the target server (if issued by a Certificate Authority that the client also trusts) enables NTLM-based authentication that guarantees both the user identity and server identity.</span></span>

#### <a name="ignoring-ntlm-based-server-identity-errors"></a><span data-ttu-id="7ccc6-146">忽略 NTLM 伺服器識別錯誤</span><span class="sxs-lookup"><span data-stu-id="7ccc6-146">Ignoring NTLM-based server identity errors</span></span>

<span data-ttu-id="7ccc6-147">如果將 SSL 憑證部署至伺服器進行 NTLM 連線並不可行，可以隱藏所產生的身分識別錯誤，方法是將該伺服器新增至 WinRM **TrustedHosts** 清單。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-147">If deploying a SSL certificate to a server for NTLM connections is infeasible, you may suppress the resulting identity errors by adding the server to the WinRM **TrustedHosts** list.</span></span> <span data-ttu-id="7ccc6-148">請注意，將伺服器名稱新增到 **TrustedHosts** 清單，不應視為任何形式對主機本身信賴的聲明，因為 NTLM 驗證通訊協定無法保證您實際上連線的主機是所要連線的主機。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-148">Please note that adding a server name to the **TrustedHosts** list should not be considered as any form of statement of the trustworthiness of the hosts themselves - as the NTLM authentication protocol cannot guarantee that you are in fact connecting to the host you are intending to connect to.</span></span> <span data-ttu-id="7ccc6-149">相反地，應考慮讓 TrustedHosts 設定成為想要隱藏錯誤 (因無法驗證伺服器身分識別而產生之錯誤) 的主機清單。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-149">Instead, you should consider the TrustedHosts setting to be the list of hosts for which you wish to suppress the error generated by being unable to verify the server's identity.</span></span>

### <a name="ongoing-communication"></a><span data-ttu-id="7ccc6-150">進行中的通訊</span><span class="sxs-lookup"><span data-stu-id="7ccc6-150">Ongoing Communication</span></span>

<span data-ttu-id="7ccc6-151">完成初始驗證之後，WinRM 會加密所有正在進行的通訊。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-151">Once initial authentication is complete, the WinRM encrypts the ongoing communication.</span></span> <span data-ttu-id="7ccc6-152">透過 HTTPS 連線時，會使用 TLS 通訊協定來交涉傳輸資料所用的加密。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-152">When connecting over HTTPS, the TLS protocol is used to negotiate the encryption used to transport data.</span></span>
<span data-ttu-id="7ccc6-153">透過 HTTP 連線時，訊息層級的加密則由所用初始驗證通訊協定來決定。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-153">When connecting over HTTP, message-level encryption is determined by initial authentication protocol used.</span></span>

- <span data-ttu-id="7ccc6-154">基本驗證不提供加密。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-154">Basic authentication provide no encryption.</span></span>
- <span data-ttu-id="7ccc6-155">NTLM 驗證使用具有 128 位元金鑰的 RC4 加密。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-155">NTLM authentication uses an RC4 cipher with a 128-bit key.</span></span>
- <span data-ttu-id="7ccc6-156">Kerberos 驗證加密由 TGS 票證中的 `etype` 決定。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-156">Kerberos authentication encryption is determined by the `etype` in the TGS ticket.</span></span> <span data-ttu-id="7ccc6-157">這是新式系統的 AES-256。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-157">This is AES-256 on modern systems.</span></span>
- <span data-ttu-id="7ccc6-158">CredSSP 加密會使用交握所交涉的 TLS 加密套件。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-158">CredSSP encryption is uses the TLS cipher suite that was negotiated in the handshake.</span></span>

## <a name="making-the-second-hop"></a><span data-ttu-id="7ccc6-159">進行第二次跳躍</span><span class="sxs-lookup"><span data-stu-id="7ccc6-159">Making the second hop</span></span>

<span data-ttu-id="7ccc6-160">PowerShell 遠端預設會使用 (如果提供) Kerberos 或 NTLM 驗證。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-160">By default, PowerShell Remoting uses Kerberos (if available) or NTLM for authentication.</span></span> <span data-ttu-id="7ccc6-161">這兩種通訊協定驗證遠端電腦時，皆不需要將認證傳送到電腦。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-161">Both of these protocols authenticate to the remote machine without sending credentials to it.</span></span> <span data-ttu-id="7ccc6-162">這是最安全的驗證方式，但因為遠端電腦並沒有使用者的認證，所以無法代替使用者存取其他電腦與服務。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-162">This is the most secure way to authenticate, but because the remote machine does not have the user's credentials, it cannot access other computers and services on the user's behalf.</span></span> <span data-ttu-id="7ccc6-163">這稱為「第二個躍點問題」。</span><span class="sxs-lookup"><span data-stu-id="7ccc6-163">This is known as the "second hop problem".</span></span>
